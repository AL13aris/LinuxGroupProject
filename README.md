# LinuxGroupProject 협업 완전 가이드

이 문서는 LinuxGroupProject의 모든 팀원이 GitHub 중앙 저장소를 기반으로 효율적으로 협업하기 위한 저장소 초기 연결 방법과 공동 작업 워크플로우를 설명합니다.

## 🌟 1. 프로젝트 시작 준비: 로컬 환경에 저장소 복제 (Clone)

모든 팀원은 Git 설치 및 SSH 키 설정을 완료해야 합니다.

### 1.1. 저장소 SSH 주소 확인 및 복사

가장 안전하고 편리한 SSH 방식을 사용하여 저장소 주소를 복사합니다.

1. **GitHub 접속**: 웹 브라우저를 열어 LinuxGroupProject 저장소 페이지에 접속합니다.
2. **주소 복사**:
   - `<> Code` (코드) 버튼을 클릭합니다.
   - **SSH** 탭을 선택하여 주소(`git@github.com:AL13aris/LinuxGroupProject.git`)를 복사합니다.

### 1.2. 로컬 환경으로 저장소 복제 (Clone)

복사한 주소를 사용하여 저장소 전체를 자신의 리눅스 컴퓨터로 다운로드합니다.

```bash
# 1. 저장소를 다운로드할 원하는 디렉터리로 이동 (예: ~/Projects)
cd ~/Projects

# 2. 복사한 SSH 주소를 사용하여 저장소 복제 명령어 실행
git clone git@github.com:AL13aris/LinuxGroupProject.git

# 3. 프로젝트 폴더로 이동하여 작업 준비
cd LinuxGroupProject
```

### 1.3. 원격 연결 상태 확인

로컬 저장소가 원격(GitHub)과 올바르게 연결되었는지 확인합니다.

```bash
# 원격 저장소 이름 확인 (기본적으로 'origin'으로 설정됨)
git remote -v

# 로컬/원격 브랜치 목록 확인
git branch -a
# (main 브랜치와 remotes/origin/main이 확인되어야 합니다.)
```

## 🚀 2. 협업 워크플로우: 관리자가 생성한 팀 브랜치에 연결하여 공동 작업

> **🚨 중요한 개발 규율**: 모든 작업은 `main` 브랜치가 아닌, 관리자가 생성하여 공유한 특정 기능 브랜치에서 진행해야 합니다.

### 2.1. 관리자: 기능 브랜치 생성 및 원격 푸시 (관리자만 수행)

관리자는 팀이 함께 작업할 브랜치(예: `feature/A`)를 생성하고 원격에 공유합니다.

```bash
# 1. 최신 main 브랜치로 이동 및 동기화
git checkout main
git pull origin main

# 2. 팀이 함께 작업할 브랜치(예: feature/A) 생성 및 이동
git checkout -b feature/A

# 3. 브랜치를 원격 저장소에 푸시하여 팀원들과 공유
git push origin feature/A
```

### 2.2. 팀원: 공유된 팀 브랜치에 연결 및 작업 시작 (팀원 수행)

팀원들은 아래 명령어를 사용하여 관리자가 만든 `feature/A` 브랜치에 접속하고 공동 작업을 시작합니다.

```bash
# 1. 원격 저장소의 최신 브랜치 정보를 가져옴
git fetch

# 2. 관리자가 만든 팀 브랜치로 전환 및 연결
git checkout feature/A
```

### 2.3. 작업 수행, 추가 (Add), 커밋 (Commit)

팀원 모두 `feature/A` 브랜치에서 코드를 수정하고, 주기적으로 변경 사항을 커밋합니다.

```bash
# (feature/A 브랜치에서 코드 수정 작업 수행)

# 1. 모든 변경 사항 스테이징 영역에 추가
git add .

# 2. 변경 사항 커밋 (커밋 메시지는 '유형: 설명' 형식 권장)
git commit -m "feat: A 기능 구현 중 일부 작업 커밋"
```

**커밋 메시지 컨벤션 예시:**
- `feat:` - 새로운 기능 추가
- `fix:` - 버그 수정
- `docs:` - 문서 수정
- `style:` - 코드 포맷팅, 세미콜론 누락 등
- `refactor:` - 코드 리팩토링
- `test:` - 테스트 코드 추가
- `chore:` - 빌드 업무, 패키지 매니저 설정 등

### 2.4. 원격 저장소로 푸시 (Push)

작업이 완료되면, 자신의 로컬 커밋 내용을 공동 작업 브랜치로 푸시하여 다른 팀원들과 공유합니다.

> **📌 충돌 방지**: 푸시하기 전에 다른 팀원이 먼저 푸시를 했는지 확인하고, 최신 내용을 먼저 가져와 병합한 후 푸시해야 합니다.

```bash
# 푸시 전 최신 내용 가져오기 (충돌 방지)
git pull origin feature/A

# 공동 작업 브랜치(feature/A)로 푸시
git push origin feature/A
```

**충돌 발생 시 해결 방법:**

```bash
# 1. 충돌이 발생한 파일을 에디터로 열어 충돌 부분 확인
# (<<<<<<, =======, >>>>>> 마커로 표시된 부분)

# 2. 충돌 부분을 수동으로 수정하여 최종 코드 결정

# 3. 수정한 파일을 스테이징
git add .

# 4. 병합 커밋 완료
git commit -m "merge: feature/A 충돌 해결"

# 5. 푸시
git push origin feature/A
```

### 2.5. Pull Request (PR) 요청 및 병합 (Merge)

모든 작업이 `feature/A` 브랜치에 통합되고 완료된 후, `main` 브랜치로의 병합을 요청합니다.

1. **PR 생성**: GitHub 웹사이트에서 `feature/A` 브랜치를 `main` 브랜치로 병합해 달라는 요청(Pull Request)을 생성합니다.
2. **코드 리뷰 요청**: 팀원들에게 최종 코드 리뷰를 요청하고 승인을 받습니다.
3. **최종 병합**: 관리자가 PR을 승인하고 `main` 브랜치로 병합(Merge)합니다.
4. **브랜치 정리**: 병합이 완료된 후 해당 기능 브랜치를 삭제합니다 (선택 사항).

```bash
# 로컬 브랜치 삭제
git branch -d feature/A

# 원격 브랜치 삭제 (관리자만)
git push origin --delete feature/A
```

## 💡 3. 다음 작업 시작: main 브랜치 동기화

한 기능에 대한 작업이 완료되면, 다음 작업을 시작하기 전에 반드시 `main` 브랜치로 돌아와 최신 변경 사항을 받아와야 합니다.

```bash
# 1. main 브랜치로 이동
git checkout main

# 2. 원격 main의 최신 내용을 로컬로 가져와 동기화
git pull origin main

# 3. (다음 작업이 있다면) 관리자에게 새로운 기능 브랜치 생성을 요청하거나,
# 관리자가 새로운 브랜치를 만들어 2.1 단계부터 다시 시작합니다.
```

## 📋 빠른 참조 가이드

### 자주 사용하는 명령어

```bash
# 현재 브랜치 및 상태 확인
git status

# 커밋 히스토리 확인
git log --oneline

# 브랜치 목록 확인
git branch -a

# 변경 사항 임시 저장 (stash)
git stash

# 임시 저장한 내용 다시 적용
git stash pop
```

## 🆘 문제 해결

문제가 발생하면 다음 순서로 확인하세요:

1. `git status`로 현재 상태 확인
2. `git log`로 최근 커밋 히스토리 확인
3. 필요시 팀 관리자나 팀원에게 도움 요청
4. Git 공식 문서 참조: https://git-scm.com/doc
